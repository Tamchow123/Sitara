# Sitara — Phased Implementation Roadmap

Companion to [PROPOSAL.md](PROPOSAL.md). Each phase lists: **Scope / Non-goals / Commands / Automated tests / Manual checkpoint / Suggested commit.** Phases are sequential; a phase's checkpoint gates the next.

Standing rules across all phases:
- Paid provider calls happen only inside `generation/providers/` wrappers, gated by `ALLOW_PROVIDER_CALLS=true` + real keys. Automated tests and CI can never spend money.
- `DEMO_MODE` is strictly zero-cost and deterministic; live generation is separately gated by `LIVE_GENERATION_ENABLED`.
- All provider model names are environment variables; none are hard-coded.
- Section references (§) point at PROPOSAL.md.

## Phase 2 — Image-model feasibility evaluation
- **Scope:**
  - *Catalogue inspection first:* inspect the CURRENT official Replicate / Black Forest Labs model catalogue (live pages, not memory or this document) and select the candidate set — one inexpensive/fast model, one balanced production-quality model, one highest-quality model, and **at least one reference-image-conditioning / image-editing-capable model (mandatory, not optional)**. Record for every candidate: exact `owner/model` identifier and version/digest where exposed, supported inputs (seed, aspect ratios, reference/edit parameters, negative-prompt support, JSON/structured prompting), current pricing with the date checked, expected and conservative per-generation cost, and provider terms (model licence, commercial use, input retention, output ownership, training use) with sources and verification dates. Unverifiable items are marked unresolved for human review — no legal conclusions.
  - *Two-stage evaluation* in `experiments/model-eval/` (standalone typed Python; pydantic-validated configs): **Stage A screening** — ~12 briefs × all candidates, one output per model and brief, to remove clearly unsuitable models; **Stage B finalists** — the top two models (human-chosen) against the full 24–36 brief matrix with multiple seeds, production portrait aspect ratio and higher-quality settings. Brief matrix covers lehenga, saree, gharara, sharara, anarkali, shalwar kameez; nikah, mehndi, baraat, walima (plus pheras/reception in finalists); modest full-sleeve styling; heavy and minimal embellishment.
  - *Three inspiration modes* compared per applicable brief: `text_only` (questionnaire-style brief), `metadata` (brief + curated inspiration metadata, no image bytes), `reference_image` (actual rights-verified reference images, only on models that support them — unsupported models are recorded as skipped, never silently ignored). References live in a manifest with rights status; anything not explicitly `verified` is rejected before any provider call.
  - *Two refinement strategies* compared: fresh regeneration (one constrained change, seed reused where supported, with the recorded caveat that seed reuse does not guarantee continuity) versus image-editing / conditioned refinement (base image + same single change + preserve-everything-else instruction), with drift comparison across requested attribute, unrelated attributes, composition, pose and garment structure.
  - *Model-specific prompt formats*: deterministic, tested formatters for natural-language editorial, structured section-based, structured JSON (only where officially supported), positive-only wording throughout, and a controlled exclusion variant only for models with genuine negative-prompt support. The exact rendered prompt is stored with every result.
  - *Hard budget enforcement*: live runs require `ALLOW_PROVIDER_CALLS=true` + `REPLICATE_API_TOKEN` + `--budget-usd` + `--confirm-live`; conservative reserve-before-call against a durable ledger (atomic file replace + process lock); reconcile after; release on pre-acceptance failures; ambiguous failures conservatively counted as spent; interrupted runs resume without re-sending or double-counting.
  - *Complete provenance* per attempted generation (run/stage/brief/model/version/prediction id/prompt payload/mode/references/refinement lineage/seed/params/dimensions/timing/status/error/costs/output hash/pricing snapshot date/git commit), plus latency and cost recording, and a versioned provider-terms snapshot (`TERMS_SNAPSHOT.md`).
  - *Blind human scoring*: contact sheets grouped by brief with anonymised candidate codes (mapping stored separately), comparisons by model / inspiration mode / prompt format / refinement strategy; 1–5 rubric (garment accuracy, cultural coherence, fabric realism, embroidery, dupatta styling, anatomy, prompt adherence, modesty/coverage adherence, reference influence, refinement consistency, overall quality) plus cultural hard-failure checks (gharara/sharara confusion, saree misrepresentation, tradition conflation, ignored coverage, dupatta errors, sexualisation, implausible construction, text/logos/designer marks, over-literal reference copying, unrelated refinement drift). No automated aesthetic scoring, no LLM judge.
  - Decision record `docs/decisions/0001-image-model.md` (starts `proposed`, completed only after scoring).
- **Non-goals:** no Django/Next.js code, no database, no Anthropic calls (prompts are hand-written/config-driven), no automation of scoring, no model selection before the experiment and human scoring complete, no scraping or unlicensed imagery, no committed generation outputs.
- **Commands:** from `experiments/model-eval/`: `python -m pytest`; `python -m model_eval.cli inspect`; `python -m model_eval.cli plan --config configs/screening.yaml --budget-usd 10`; `python -m model_eval.cli run --config configs/screening.yaml --dry-run --budget-usd 10`; live (all four gates): `ALLOW_PROVIDER_CALLS=true REPLICATE_API_TOKEN=... python -m model_eval.cli run --config configs/screening.yaml --budget-usd 10 --confirm-live`; then `contact-sheet`, `scoring-sheet`, `budget-status` subcommands per run.
- **Automated tests:** config validation (malformed candidates rejected); deterministic matrix expansion and prompt rendering; screening/finalist matrices distinct; required garment/ceremony/modesty/embellishment coverage; capability-driven format selection; unsupported reference runs recorded as skips; unverified references rejected; zero network calls proven for dry-run and for each missing live gate; reservation-before-provider-call ordering; reconciliation capped at the ceiling; resume without duplicates; no silent output overwrites; full provenance field coverage; secret redaction; blind-by-default contact sheets. All provider calls mocked — CI never reaches Replicate.
- **Manual checkpoint:** all rubric dimensions and hard-failure checks scored for every model × brief cell in both stages (blind); the mode and refinement comparisons reviewed; decision record completed naming default + fast model, the inspiration-influence recommendation and refinement-strategy recommendation with drift findings; total spend within budget and reconciled in the ledgers.
- **Commit:** `experiment: add budget-controlled FLUX model feasibility evaluation`

## Phase 3B — Session authentication *(delivered; inserted, no renumbering)*
- **Scope:** Django database sessions + `authenticate()/login()/logout()` + CSRF as the only auth mechanism; endpoints `GET /api/v1/auth/csrf/`, `POST /api/v1/auth/register/`, `POST /api/v1/auth/login/`, `POST /api/v1/auth/logout/`, `GET /api/v1/auth/me/`; project cookie names `sitara_sessionid`/`sitara_csrftoken` (HttpOnly session, SameSite=Lax, Secure in production); JSON CSRF failure view; 12-char-minimum password policy via Django validators; Redis fixed-window auth rate limits (HMAC-hashed identifiers, fail-closed 503); same-origin Next.js `/api/:path*` rewrite via server-only `API_INTERNAL_BASE_URL` (no `NEXT_PUBLIC_*` backend host); frontend `/login`, `/register`, `/account` with in-memory CSRF client and cookie-presence middleware (navigation-only). Accounts are an **optional** capability — this revises the original "no accounts" MVP non-goal (ADR 0003).
- **Non-goals:** email verification, password reset/change, email change, account deletion, OAuth, MFA, magic links, JWT, refresh tokens, profiles, questionnaire/design models, guest design claiming, uploads, provider calls, email delivery, deployment. Public production registration is not feature-complete until email verification + password recovery are designed.
- **Commands:** `docker compose up -d`; `docker compose exec api pytest`; `docker compose exec web npm test -- --run`; manual flow via `http://localhost:3001/api/v1/auth/...`.
- **Automated tests:** CSRF enforcement with `enforce_csrf_checks=True`; generic invalid-credential/duplicate-registration responses; session-key rotation on login; logout session invalidation; rate-limit 429/`Retry-After`/hashed keys/503 outage; `Cache-Control: no-store`; log-safety capture; frontend CSRF bootstrap/retry-once, open-redirect rejection, storage-free credentials.
- **Manual checkpoint:** register → me → logout through the Next.js origin; HttpOnly `sitara_sessionid` verified; no credentials in browser storage.
- **Commit:** `feat(auth): add secure session authentication`

## Phase 4 — Minimal database and session foundation *(delivered)*
- **Scope:** DesignSession, Design, DesignVersion, GenerationAttempt models + migrations + admin registration; **dual ownership** (ADR 0004): anonymous workspaces via **Django's standard session framework** (DesignSession created lazily; its UUID stored in Django session data under `sitara_design_session_id` — no raw session key in any domain table) **and** authenticated user ownership (nullable `user` FK; an anonymous workspace is automatically claimed by the user on the next design API request after login, surviving Django's session-key rotation); design API `GET/POST /api/v1/designs/` + `GET/PATCH /api/v1/designs/<uuid>/` (DRF, `AllowAny` + mandatory ownership filtering, Django `csrf_protect` on unsafe methods, title-only writes, unknown fields rejected, `Cache-Control: no-store`); ownership helpers (inaccessible design → 404, never 403). DesignVersion: positive-integer `version_number` with a unique `(design, version_number)` constraint and `CHECK (version_number > 0)`; `MAX_DESIGN_VERSIONS=2` enforced in a row-locking application service only.
- **Non-goals:** no questionnaire content, no catalogue, no generation logic, no retention purge, no custom auth cookie of any kind, no deletion/sharing endpoints, no frontend design pages, no manual claim endpoint.
- **Commands:** `docker compose exec api python manage.py makemigrations` then `... migrate`; `docker compose exec api pytest`.
- **Automated tests:** model constraints (idempotency-key uniqueness, `(design, version_number)` uniqueness, positive version_number, cascades, non-shared JSON defaults); application-level rejection of a version beyond `MAX_DESIGN_VERSIONS` incl. a real concurrency test; CSRF enforced on unsafe endpoints (`enforce_csrf_checks=True`); the twelve anonymous→authenticated ownership flows (login promotion, cross-browser access, shared-browser account switching, logout, 404-not-403); no DesignSession identifier or session key in any response.
- **Manual checkpoint:** anonymous design create/list via the Next.js origin (observing the CSRF flow); second browser session gets 404; register in the first session and confirm the design appears in the authenticated list; inspect in Django admin.
- **Commit:** `feat(designs): add private design ownership foundation`

## Phase 5A — Versioned questionnaire taxonomy *(delivered; split from Phase 5, no renumbering)*
- **Scope:** `QuestionnaireVersion` model (UUID PK, globally unique positive `version`, draft/active/retired status, JSON `schema`, staff `created_by`/`activated_by` with SET_NULL) with named DB constraints: `version > 0`, valid status, and a PostgreSQL **partial unique constraint enforcing at most one active version**; pure-Python schema-format validator (question types `single_choice`/`multi_choice`/`text` only; bounded constraints incl. mandatory text caps; allowlisted compatibility rules `equals|in|not_in` → `show|hide|require|restrict_options`; strict machine-id pattern; no expression engine — ADR 0005); locking, transactional `activate_questionnaire_version` service (draft-only activation, full validation, previous active retired in the same transaction); published versions immutable (version + schema frozen once active/retired, no admin deletion of active); public `GET /api/v1/questionnaire/active/` (AllowAny, identity-free — creates no session and no DesignSession; exactly `{id, version, schema}`; `Cache-Control: no-store`; safe `503 questionnaire_unavailable` for missing **or corrupt** active schema, logging only version id + exception type); admin workflow (drafts editable, activation only via action, safe error messages); seeded `questionnaire_v1` fixture with the full garment/ceremony/colour/fabric/embellishment/coverage/draping taxonomy **including machine-readable validation/compatibility constraints per question** (the source the frontend later derives Zod from, and the same constraints Django's authoritative validator applies), with fixture-integrity and cultural-distinction tests (gharara ≠ sharara, saree drape ≠ lehenga styling, designer-name denylist).
- **Non-goals:** no inspiration catalogue (Phase 5B), no answer submission/persistence, no `Design`↔`QuestionnaireVersion` linkage, no frontend wizard or Zod, no OpenAPI, no provider calls, no new dependencies.
- **Commands:** `docker compose exec api python manage.py migrate`; `docker compose exec api python manage.py loaddata questionnaire_v1`; `docker compose exec api pytest`.
- **Automated tests:** DB constraints (positive/unique version, valid status, single active); schema-validator success and rejection matrix (malformed top level, duplicate ids, unsupported types, invalid bounds, dangling rule references, oversized schemas); activation atomicity incl. failed activation leaving the prior active version untouched; immutability of published versions; endpoint shape/privacy/no-store/no-session tests; fixture integrity and cultural coverage.
- **Manual checkpoint:** `GET /api/v1/questionnaire/active/` through the Next.js origin returns version 1 with all six garments and six ceremonies and sets no cookie; admin draft → activate action → previous version retired → endpoint serves the new version.
- **Commit:** `feat(questionnaire): add versioned taxonomy schema`

## Phase 8 — Structured DesignSpec generation *(delivered — integrity hardening pending hosted CI)*
- **Scope:** Pydantic `DesignSpec` model; JSON schema derivation; Anthropic wrapper using structured outputs (model from `ANTHROPIC_MODEL`); system prompt as a source-controlled constant with `SPEC_TEMPLATE_VERSION`; input-safety layer (caps, denylist incl. designer names, delimited untrusted free text); Django-side re-validation; recorded-fixture responses for tests; one manual budgeted live invocation command.
- **Non-goals:** no image generation, no Celery (synchronous management command for now), no refinement.
- **Commands:** `pytest backend/` (fixtures only); offline: `python manage.py generate_spec --design <id> --fixture valid` (zero network); manual live (paid, requires separate spend approval): `DEMO_MODE=false ALLOW_PAID_AI_CALLS=true ANTHROPIC_API_KEY=… python manage.py generate_spec --design <id> --confirm-live` (run inside the backend container). The obsolete `ALLOW_PROVIDER_CALLS` variable is not used.
- **Automated tests:** fixture-driven spec validation (valid spec persists; invalid spec → one retry → hard fail); denylist blocks pre-spend; wrapper raises without env gate (**asserting zero live calls in CI**); Pydantic re-validation rejects a deliberately malformed fixture.
- **Manual checkpoint:** one budgeted live run produces a valid, culturally coherent DesignSpec from a real questionnaire submission; review output quality by hand.
- **Commit:** `feat(generation): schema-constrained DesignSpec via Anthropic structured outputs`

> **Delivered (ADR 0009)** as two commits — `feat(generation): add versioned DesignSpec contract` and `feat(generation): add gated Anthropic spec generation`. A strict Pydantic v2 `DesignSpec` (bounded, literal `schema_version`, exact `source_selections` echo) drives a deterministic committed JSON Schema (`sitara/generation/schemas/design_spec_v1.json`, byte-identity guarded). Generated output and free text are scanned by a conservative designer/brand denylist (NFKC/case/punctuation-robust, generic categories). `pydantic==2.13.4` and `anthropic==0.116.0` added (hash-locked). Generation uses `beta.messages.parse(output_format=DesignSpec)` with `max_retries=0` and at most **two** requests (one retry only on invalid output); Django re-validates and verifies `source_selections` exactly. `generate_design_spec_for_design` performs every pre-spend gate first, takes a non-blocking advisory lock keyed by the Design UUID, and persists one `DesignVersion` with narrow provenance (all-or-none + positive-token DB constraints) — never a prompt, response or key. Capability gates are explicit (`STRUCTURED_DESIGN_PROVIDER_IMPLEMENTED=True`, `IMAGE_PROVIDER_IMPLEMENTED=False`); public `generation_is_available()` stays False. The `generate_spec` command offers offline `--fixture` mode and a live mode gated by `--confirm-live` + open env gates. No API, frontend, Celery or image work. A follow-up commit — `fix(generation): harden DesignSpec generation integrity` — centralised the live gate (both env gates **plus** a non-empty key and a bound-fitting model, via one `structured_design_generation_is_available()` used by the command too), validates the source-selection contract pre-spend (`unsupported_questionnaire_contract`), rejects stale inputs (a snapshot + completion re-check before persistence, no transaction held across the call → `DesignChangedDuringGeneration`), aggregates token usage honestly across both attempts, hardens the scanner (underscore-safe normalisation, scope-aware claim negation) and adds DesignSpec semantic invariants (required caveats; regional-direction consistency). The **live** quality checkpoint requires separate, explicit spend approval and was **not** run; Phase 8 is not marked fully delivered until this hardening lands with hosted CI green.

## Phase 16 — Security and live-generation cost controls
- **Scope:** `LIVE_GENERATION_ENABLED` flag with `503 live_generation_disabled` when off; per-session + per-IP DRF/Redis throttles and daily generation count limit (`429 generation_limit_reached`); **hard daily cost ceiling with atomic reserve-before-spend**: compute a conservative maximum estimated cost for the call, atomically reserve it in Redis via a Lua script (or equivalent atomic transaction) *before* provider invocation, reject with `503 live_generation_budget_exhausted` if the reservation would exceed the ceiling, reconcile the reservation down to the estimated actual cost after the call, and release unused reservation on failure paths where no spend occurred — no check-then-increment race between workers. - **Non-goals:** no BudgetWindow table unless the Redis mechanism proves insufficient; no spend dashboard (ad-hoc GenerationAttempt queries only).

- **Automated tests:** disabled live generation → 503 + code; limits → 429 + code; ceiling exhaustion → `503 live_generation_budget_exhausted` and showcase degrade, never an unhandled 5xx; **concurrency test: N parallel reservation attempts against a ceiling that fits N−1 admit exactly N−1** (proves atomicity); reservation reconciled after success and released after pre-spend failure; purge deletes rows *and* storage objects; stuck job (>10 min) reconciled to failed; throttle counters keyed by session and hashed IP.
